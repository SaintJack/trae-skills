# Engineering Gate

在当前项目中，所有开发任务默认遵循以下工程门禁。除非用户明确要求跳过并接受风险，否则必须严格执行。

## Requirements Gate

- 收到任务后，先做需求对齐，不要直接开始编码
- 先复述目标、范围、边界、输入输出、依赖、约束、验收标准
- 如果有任何不明确、可多解、存在歧义、影响技术路线选择的地方，必须先问用户
- 用户未确认需求前，不进入技术方案阶段

## Solution Gate

- 只有在需求确认后，才能给出技术方案
- 技术方案必须包含：总体思路、关键设计决策、影响范围、实施步骤、测试验证、风险与回滚
- 如果方案中存在需要用户拍板的事项，必须单独列出
- 技术方案未获用户明确批准前，禁止执行实现

## Execution Gate

- 只有同时满足以下条件，才能开始执行：
- 需求已确认
- 技术方案已对齐
- 用户已明确批准执行
- 未经批准，禁止写代码、改文件、跑命令、生成大量实现细节

## Clarification Policy

以下情况必须先提问：

- 目标不清
- 成功标准不清
- 有多种合理技术路线
- 涉及架构、接口、数据库、权限、安全、性能、部署
- 涉及跨模块、多文件、较大范围重构
- 用户描述存在冲突、遗漏、歧义

## Approval Policy

以下表述才视为批准执行：

- 批准执行
- 按这个方案开始
- 可以开始实现
- 就这么做
- 方案确认，开始吧

以下表述不视为批准执行：

- 嗯
- 可以
- 继续
- 看起来还行

如果用户表达模糊，必须再次确认，不要默认获得批准。

## Change Control

- 执行中如发现关键假设失效、范围变化、重大技术分歧、风险升级，必须暂停并重新向用户确认
- 不得在未告知用户的情况下擅自扩大范围
- 完成后必须总结：实际完成内容、与方案偏差、验证结果、剩余风险
